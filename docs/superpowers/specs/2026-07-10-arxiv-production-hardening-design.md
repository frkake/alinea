# arXiv 取り込み・翻訳・記事・日本語 PDF 完全性強化 設計

日付: 2026-07-10  
状態: 承認済み  
対象: arXiv 拡張機能取り込みから Viewer、記事、図表、日本語 PDF まで

## 1. 背景

未登録の arXiv 論文 20 本を実サービス構成で E2E 検証したところ、フロントエンドからの保存は 20/20、原文 PDF 表示は 20/20 成功した一方、次の不完全性が確認された。

- 取り込みは 15/20 成功。5 本は LaTeX 解析失敗後の HTML 404 で停止し、保存済み原文 PDF を解析しなかった。
- 1 本は LaTeX ラッパー内の埋め込み PDF ファイル名だけを本文として構造化し、成功扱いになった。
- 完了扱いの翻訳 15 本中 9 本に未翻訳ユニットが残った。
- 意味のある本文を得た 14 本すべてで、本文、図表・参考文献、または記事のいずれかに生の LaTeX コマンドが表示された。
- 図を含む 12 本のうち、全図が表示できたのは 5 本だった。
- 記事ジョブは 20/20 成功したが、取り込み失敗 5 本と本文誤解析 1 本は要旨中心の劣化記事だった。全体概要図は 20/20 失敗した。
- 日本語 PDF は 3/20 のみ生成され、3 本とも英語本文が残った。完全な日本語 PDF は 0/20 だった。
- デスクトップ表示の横はみ出し・クリップは検出されなかった。

本設計は、これらを論文固有の ID、タイトル、著者、個別ソース断片に依存せず修正する。

## 2. 目標

### 2.1 機能目標

1. 拡張機能から保存した arXiv 論文を、LaTeX、HTML、PDF テキスト、PDF OCR のいずれかで必ず構造化する。
2. 要旨・本文・付録・脚注・図表キャプション・表セルを翻訳し、未訳を残した状態を完了扱いにしない。
3. 表示可能な本文、図表、記事から生の LaTeX 制御綴を排除する。
4. 原稿内で参照される全図を有効な画像資産として保存し、フロントエンドから表示する。
5. 検証済み本文から記事、全体概要図、解説図を生成する。
6. 全ソース形式から、全訳範囲を含む日本語 PDF を生成する。
7. すべての操作と再試行をフロントエンドから実行可能にする。
8. 再取り込み時に注釈、読書状態、コレクション所属などのユーザーデータを維持する。

### 2.2 合格条件

今回の 20 本を通常の製品経路で再処理し、20/20 で以下を満たすことを最終合格条件とする。

- 検証済みの意味のある本文が存在する。
- 翻訳対象ユニットに空文字、未訳、原文フォールバック、壊れたプレースホルダーがない。
- 訳文、図表・参考文献パネル、記事に生の LaTeX 制御綴が表示されない。
- 構造化された図の全件が、有効な画像として HTTP 200 かつ正の画像寸法で表示される。
- 構造化された表の全件が DOM に存在する。
- 記事本文、全体概要図、必要な解説図が表示される。
- 原文 PDF と日本語 PDF が表示・ダウンロードできる。
- 日本語 PDF に翻訳対象の英語本文が残らず、生 LaTeX、空ページ、ページ境界違反がない。
- 1440x900 の Viewer 全モードで横方向のクリップ・はみ出しがない。

## 3. 非目標

- 参考文献の著者名、論文名、誌名などの書誌情報は翻訳しない。
- 数式、コード、識別子、URL を自然言語として翻訳しない。
- 原論文の誤りや、LLM 翻訳の学術的内容そのものを校閲する機能は追加しない。
- 特定の arXiv ID、タイトル、著者、既知の論文固有マクロを製品コードに登録しない。
- UI を刷新しない。既存操作を保ちながら状態表示と再試行を明確化する。

## 4. 設計原則

### 4.1 構造による判定

フォールバックや補正は、ファイル形式、構文木、資産の存在、PDF テキスト量、ブロック対応率などの構造的シグナルだけで判断する。論文の識別情報は判断に使用しない。

### 4.2 成功より完全性を優先

不完全な本文、未訳、欠損図、英語本文を含む日本語 PDF、素材不足記事を成功扱いにしない。利用可能な中間成果は保持しても、ジョブ状態と UI は未完了または失敗を明示する。

### 4.3 決定的フォールバック

外部プロバイダーが失敗しても、原文 PDF 解析、数式の可読テキスト化、概要 SVG、構造化日本語 PDF など、ローカルで決定的に生成できる経路を持つ。

## 5. アーキテクチャ

### 5.1 原本取得と候補パイプライン

arXiv 取り込み開始時に原文 PDF を必須原本として取得・保存する。API が先行取得済みの場合は同じ `SourceAsset` を再利用する。LaTeX と HTML の取得は原文 PDF の有無に影響させない。

構造化候補を次の順で評価する。

1. arXiv LaTeX
2. arXiv HTML
3. 原文 PDF のテキストレイヤー
4. 原文 PDF の OCR

各候補は共通の `DocumentCompleteness` 判定を通過した場合だけ採用する。候補が取得できても完全性を満たさなければ、診断情報を保存して次候補へ進む。

### 5.2 文書完全性ゲート

完全性判定は次を検査する。

- 翻訳対象となる本文文字列が存在する。
- PDF テキストレイヤーが 1,000 文字以上ある場合、構造化本文がその可視テキストの 35% 以上を保持する。
- セクションまたは複数の意味のある段落が存在する。
- 構造化結果がファイル名だけ、`\includepdf` だけ、または単一の埋め込み PDF 参照だけではない。
- LaTeX が埋め込み PDF を本文として使用する場合、その PDF を次の PDF 候補として解析する。
- 図ブロック数、表ブロック数、翻訳対象ブロック数を診断値として記録する。

短い論文を排除しないため、絶対文字数だけでは失敗判定しない。PDF との比率、ブロック型、ラッパー構造を組み合わせる。

### 5.3 LaTeX 正規化と数式表示

LaTeX パーサーは原稿内の `\newcommand`、`\def`、`\DeclareMathOperator` などから文書ローカルのマクロ表を構築する。保存する数式には、可能な限りマクロを展開した標準 LaTeX を使用する。

本文側のスタイル・組版命令はトークン単位で処理する。

- スタイルラッパーは内容を保持して命令だけ除去する。
- `\@...` などの内部命令、制御文字、寸法指定は可視本文に出さない。
- 数式中の未定義マクロは、引数を保持した識別子または演算子へ一般化する。
- KaTeX が失敗した場合は生 LaTeX を表示せず、標準記号へ変換した可読なプレーン数式を表示する。
- 文書ローカルのマクロ表は Viewer と記事レンダラーの両方へ渡す。

この処理はマクロ名の固定リストではなく、定義形とトークン種別に基づく。

### 5.4 図表資産

LaTeX 図資産の解決は次を行う。

- メイン `.tex` からの相対パス解決
- 拡張子省略時の `.pdf`、`.png`、`.jpg`、`.jpeg`、`.eps`、`.ps`、`.svg` 探索
- `\graphicspath` の適用
- 条件分岐やマクロに包まれた `\includegraphics` 引数の展開
- PDF/EPS/PS/SVG の PNG 化
- デコード後の幅・高さ検証

HTML 図資産は、絶対 URL、文書 URL 相対、arXiv バージョン相対を正規化して取得する。

ソース資産が解決できない場合は、原文 PDF の画像・描画領域を抽出し、図の順序、キャプション、ページ上の近接関係から対応付けて切り出す。対応付けできない図が残る場合、その候補文書は完全性ゲートを通過しない。

表は構造化セルを優先し、構造化不能な場合のみ PDF 領域画像へフォールバックする。

### 5.5 翻訳完全性

翻訳対象は要旨、本文、見出し、付録、脚注、図表キャプション、表セルである。参考文献の書誌情報、数式、コード、URL は対象外とする。

翻訳セット作成時に期待ブロック ID 集合を確定する。各ユニットはプレースホルダー、数値、参照、数式を検証し、失敗時は同一出力の再利用ではなく修復プロンプトで再試行する。プロバイダーチェーンを使い切っても検証を通らない場合、翻訳セットは `complete` にしない。

完了条件は次の集合一致である。

`期待ブロック ID = 検証済みで表示可能な翻訳ユニットのブロック ID`

### 5.6 記事と図

記事生成 API は完全性ゲートを通過したリビジョンだけを受け付ける。本文が未取得の場合は素材不足記事を保存せず、再取り込み可能なエラーを返す。

記事入力は正規化済みの構造化文書から構築し、生 LaTeX や未解決資産キーを含めない。

全体概要図は次の二段構成とする。

1. LLM の構造化 DSL 生成
2. 失敗時、記事の見出し・要点・根拠アンカーから決定的な 3 カード SVG を生成

決定的フォールバックも通常の `OverviewFigure` として保存し、UI 契約を変えない。解説図は画像生成失敗時に壊れた画像ブロックを残さず、再試行可能な状態を明示する。

### 5.7 日本語 PDF

二つのレンダラーを用意する。

1. LaTeX 原稿保持レンダラー: 翻訳ブロックを原稿へ 100% 対応付けできる場合に使用する。
2. 構造化文書レンダラー: 対応付け不能、HTML、PDF、OCR ソースで使用する。共通 LuaLaTeX テンプレートに見出し、段落、数式、図、表、脚注、参考文献を配置する。

付録を含む全翻訳対象ブロックをレンダーマニフェストへ記録する。日本語 PDF は次の検証をすべて通過した場合だけ公開する。

- 期待翻訳ブロックとレンダーマニフェストが一致する。
- PDF が開け、1 ページ以上ある。
- 日本語テキストが存在する。
- 翻訳対象ブロックに対応する英語原文フォールバックがない。
- テキスト、画像、リンクがページ境界外にない。
- 可視テキストに未処理の LaTeX 制御綴がない。

参考文献、数式、コード、固有名詞の英語は英語本文残存判定から除外する。

### 5.8 フロントエンド状態

既存の操作経路を維持し、次を追加・修正する。

- 取り込み、翻訳、記事、日本語 PDF の不完全状態と理由を表示する。
- フロントエンドから同一アイテムの再取り込み、翻訳再試行、記事再生成を実行できる。
- 日本語 PDF ボタンは検証済み資産がある場合だけ有効化する。
- 壊れた画像要素を表示せず、資産再生成中または失敗状態を表示する。
- 完了イベント受信後、Viewer の本文、図表、記事、PDF 可用性を再取得する。

## 6. エラー処理と観測性

候補ごとの失敗は `jobs.log` に、方式、段階、機械可読コード、詳細を記録する。最終エラーは試した候補と各失敗理由を含める。

主要コードは次の分類とする。

- `source_candidate_unavailable`
- `document_incomplete`
- `embedded_pdf_wrapper`
- `figure_asset_unresolved`
- `translation_incomplete`
- `article_source_incomplete`
- `translated_pdf_incomplete`

外部取得・LLM・画像生成は既存のリトライ方針に従う。構文・完全性エラーは同じ入力の無条件再試行をせず、次候補または修復経路へ進む。

## 7. 冪等性と既存データ再処理

再取り込みは新しい `DocumentRevision` と `TranslationSet` を生成し、完全性検証後に現行リビジョンへ昇格する。失敗時は従来の現行リビジョンを保持する。

リビジョン昇格時は既存のリアンカー処理を使い、注釈、読書位置、ブックマークを移行する。LibraryItem、コレクション、タグ、読書状態は再作成しない。

記事は新しい検証済みリビジョンに対して通常の再生成 API を使用する。旧記事、概要図、解説図は新生成物のコミット成功後に置換する。

今回の 20 本は QA ハーネス側のテストデータとして指定してよいが、製品コード、設定、マイグレーション、プロンプトには ID やタイトルを含めない。

## 8. テスト戦略

### 8.1 TDD 回帰フィクスチャ

実論文をコピーせず、失敗形状を再現する最小の合成フィクスチャを作る。

- LaTeX 構文失敗 + HTML 404 + 有効な PDF
- 本文を埋め込み PDF 一枚だけで構成する LaTeX ラッパー
- 文書ローカルマクロ、内部 `\@` 命令、未定義数式マクロ
- 条件分岐された画像、拡張子なし画像、EPS/PDF 図
- HTML 相対画像 URL
- 一部ユニットだけ未訳またはプレースホルダー破損
- LaTeX への翻訳対応付けが不完全な文書
- 構造化出力を拒否する概要図プロバイダー

各修正は、失敗テストを確認してから最小実装を行う。

### 8.2 単体試験

- 候補選択と文書完全性判定
- マクロ収集・展開・可視テキスト正規化
- 図資産候補解決・変換・検証
- 翻訳集合の完全性判定と修復再試行
- 記事入力の完全性ガード
- 決定的概要 SVG
- 二種類の日本語 PDF レンダラーと完全性検証
- UI の失敗理由、再試行、PDF ボタン状態

### 8.3 統合試験

- LaTeX、HTML、PDF テキスト、PDF OCR の各成功経路
- 候補失敗から次候補への遷移
- チェックポイントからの再開と同一ジョブの冪等性
- 再取り込み後のリビジョン昇格と注釈リアンカー
- 翻訳、記事、概要図、日本語 PDF までの一連処理

### 8.4 E2E

拡張機能をロードした実ブラウザで、20 本すべてについて次を操作・検査する。

1. arXiv ページから保存
2. 訳文、対訳、原文の切り替え
3. 図表・参考文献パネル
4. 記事生成と概要図
5. 原文 PDF と日本語 PDF
6. DOM 上の生 LaTeX、画像ロード、表件数、レイアウト
7. 日本語 PDF のテキスト、レンダーマニフェスト、ページ境界

## 9. ロールアウト

1. 合成回帰テストを追加する。
2. 原本取得・候補選択・完全性ゲートを実装する。
3. LaTeX 正規化と図資産処理を実装する。
4. 翻訳・記事・概要図の完全性を実装する。
5. 構造化日本語 PDF と検証を実装する。
6. UI の状態・再試行を実装する。
7. 単体・統合・既存 E2E を通す。
8. 今回の 20 本を通常経路で再取り込み・再生成する。
9. 20/20 の合格条件を満たすまで、論文固有分岐を加えず失敗形状を一般化して修正する。

## 10. リスクと対策

- OCR は遅く誤認識を含む: テキストレイヤーを優先し、OCR はページ単位の最終候補に限定する。
- PDF 図対応付けが曖昧: 図順、キャプション近接、画像領域を併用し、曖昧な場合は成功扱いにしない。
- 厳格な翻訳検証で完了率が下がる: 修復プロンプトとプロバイダーチェーンを使い、失敗を隠さない。
- 構造化 PDF は原稿と改ページが異なる: 内容完全性を優先し、原稿保持レンダラーを第一候補にする。
- 再処理がユーザーデータを壊す: 新リビジョンを先に完成させ、検証後の原子的昇格と既存リアンカーを使う。

## 11. 完了の定義

単体・統合・E2E の全試験が成功し、今回の 20 本が §2.2 の全条件を満たし、製品コードに対象論文の ID、タイトル、著者、論文固有条件が存在しないことを確認した時点で完了とする。
