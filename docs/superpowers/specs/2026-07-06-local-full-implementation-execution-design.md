# Alinea — ローカル完全実装 実行設計

> 本書は**実行設計(execution design)**である。プロダクト仕様は docs/00〜12、実装計画は plans/00〜13、見た目の正は確定デザイン16画面(`論文読解システム デザイン.dc.html`)であり、それらは既に確定済み。本書はそれらを**実際にビルド・起動・テストまで通してローカルで完璧に動く状態**へ到達させるための順序・検証ゲート・並列化戦略・完成判定を確定する。
>
> 作成日: 2026-07-06 / 承認: ユーザー(2026-07-06、AskUserQuestion + 「Chrome拡張も」の明示)

## 1. ゴール(この実装が満たすべきこと)

- **フロントエンドの完璧さ**: 確定デザイン16画面に忠実な UI が、実データ(Rectified Flow arXiv:2209.03003 シード)で動作する。トークン値・レイアウト px・文言はデザイン抽出値の逐語。
- **バックエンドの完璧な動作**: 取り込み→パース→翻訳→チャット→検索→記事→共有→語彙の全パイプラインが、**API キー無しでも**決定的に動作し(Fake/モック経由)、**BYOK/運営キーを入れれば実プロバイダで**動作する。P3(黙って壊れない)に従い、キー未設定時は原文フォールバック+失敗理由+再試行で**可視的に縮退**する。
- **Chrome/Edge 拡張のローカル完全動作**: WXT で unpacked ビルドし、`chrome://extensions` にロードして arXiv abs で保存→進捗→ビューア到達まで実機検証する。ストア申請(外部審査)のみローカル対象外。
- **グローバルホスティングは今回行わない**が、`docs/deployment.md` に手順書を残す(plans/01 §8 準拠)。

## 2. 実行方針の確定事項(ユーザー決定)

| # | 論点 | 決定 |
|---|---|---|
| E1 | ビルド順序 | **M0 を完動→M1→M2 の段階完遂**。各マイルストーンは「実際にビルド・起動・テストが green」になって初めて done。 |
| E2 | LLM 既定動作 | **実プロバイダを既定ルーティング**にする(5社テキスト+3社画像アダプタを仕様どおり全実装)。ただし**決定的 Fake プロバイダ + モックサーバ**(M0-08/M0-09)を自動テスト・E2E の駆動に使い、**キー無しでビルド・テストが green** になることを保証する。キー未設定の実行時は P3 準拠で可視的に縮退。 |
| E3 | 拡張スコープ | Chrome + Edge を**ローカルで unpacked ビルド+実機検証**まで含める。ストア申請(M0-37)は手順書に記載のみ。 |

## 3. 完成判定(「完璧に動く」の具体的定義)

各マイルストーンで以下を**実際に実行し、実出力を確認**する(「動くはず」で済ませない)。

**共通の検証ゲート(マイルストーンごと)**:

1. `docker compose up -d --wait` — db(PGroonga)/redis/minio/mailpit が healthy。
2. `uv sync --all-packages` / `pnpm install` が成功。
3. `alembic upgrade head`(M0 は全DDL一括、以後は逸脱修正のみ)。
4. `python -m alinea_api.seed --sample rectified-flow` が投入成功。
5. `pnpm turbo build lint typecheck` が green(tokens/api-client 生成含む)。
6. `pnpm dev` で web(3000)+api(8000)+worker(arq)が起動しヘルスチェック通過。
7. そのマイルストーンの **pytest / Vitest スイートが green**。
8. そのマイルストーンの **Playwright E2E が green**(LLM はスタブ、Chromium 1440×900)。
9. 拡張: `pnpm --filter @alinea/extension build` 成功 + unpacked ロードで手動シナリオ通過(M0)。

**マイルストーン別 DoD**(docs/10 §2〜4・plans/13):

- **M0**: 拡張から arXiv 論文を保存すると1分以内に読み始められ、数式・図表が崩れない日本語で読み通せ、選択質問に根拠つきで答えが返り、読みかけ位置が保存される。(AC-10-01〜04 + 取り込み経路が拡張のみ)
- **M1**: 朝のトリアージ+読了記録が回り、横断検索(日英クロス・ヒット源明示)で想起でき、arXiv にない PDF も品質 B で読める。(AC-10-05〜07 + リビジョン昇格は提案のみ + モバイル閲覧)
- **M2**: 記事モード(概要図 SVG・版管理)で読み返せ、コレクションを URL 共有でき、語彙が SRS 復習に現れ、LaTeX 由来の品質 A 主経路で読める。(AC-10-08〜11 + 記事公開 UI 不在)

M3 は明示的にスコープ外(docs/10 §5)。

## 4. 実装アーキテクチャの層構造(依存順 = 実装順の骨格)

plans/13 の WBS を正としつつ、実行者(私)は以下の**load-bearing な基盤を先に自分で構築**し、インターフェースを凍結してから並列化する。

**基盤コア(直列・自分で構築)**:

```
モノレポ骨格(M0-01) → docker-compose(M0-03) → packages/tokens(M0-05)
  → DB モデル+全DDL(M0-06) → py-core ドメイン基盤(M0-07)
  → packages/llm 抽象化層+Fake(M0-08) → モックサーバ(M0-09)
  → API 共通基盤(M0-10) → 認証(M0-11) → ジョブ基盤+SSE(M0-12) → LLM ルーティング/BYOK(M0-13)
```

**パイプラインコア(直列が主・クリティカルパス)**:

```
arXiv 解決(M0-14) → HTML パーサ(M0-15) → プレースホルダ(M0-16)
  → 翻訳パイプライン(M0-17) → 取り込みステートマシン(M0-18)
```

**インターフェース凍結後に並列化(background workflow で fan-out)**:

- API 面: ingest/papers, viewer/translations, chat, library, settings(M0-19〜23)→ api-client 生成(M0-24)
- FE 面: app シェル(M0-26)→ UI 共通17種(M0-27)→ ビューアシェル(M0-28)→ 各表示モード(M0-29〜31)→ ライブラリ/設定画面(M0-32〜33)
- 拡張面: 拡張基盤(M0-34)→ 3状態ポップアップ(M0-35)→ バッジ(M0-36)
- シード(M0-25)

**統合と検証は実リポジトリで私が実施**(workflow の出力を鵜呑みにせず、実際にビルド・テストを回す)。

## 5. 並列化とオーケストレーション戦略

- **基盤コアとパイプラインコアは私が直接実装**する。ここは型・スキーマ・アンカー処理・ルーティングという全機能が乗る土台であり、乖離が全体に波及するため(docs/01「本書のモデルが崩れると全機能が崩れる」)。
- **真に独立な作業**(UI コンポーネント群、5社+3社のプロバイダアダプタ、画面別実装、テストフィクスチャ)は、インターフェース凍結後に **Workflow で fan-out** し、各成果を実リポジトリへ統合後に**私自身がビルド・テストで検証**する。
- 検証は**敵対的に**行う: 生成物は「動くと主張された」だけでは信用せず、実際に import・ビルド・実行して確認する。
- 各マイルストーン境界で**チェックポイント**を打ち、実際に動く状態でユーザーへ進捗報告する(半統合の山を作らない)。

## 6. LLM の二層構成(E2 の具体化)

plans/04・plans/12 の設計をそのまま使う:

- `alinea_llm` に 5社テキスト(OpenAI/Anthropic/Google/DeepSeek/xAI)+3社画像(OpenAI/Google/xAI)アダプタを**全実装**。ルーティング・フォールバック・structured output・count_tokens・BYOK(Fernet)・クォータ計測まで。
- `FakeLLMProvider` / `FakeImageProvider` と `mock_server.py`(5社+arXiv+GitHub/YouTube 相当、ポート8090)を実装。`ALINEA_*_BASE_URL` 環境変数で実エンドポイントを差し替え、**キー無しの CI・ローカルテストが決定的に green** になる。
- 既定ルーティングは実プロバイダ(plans/01 §8.4 のシード表)。運用キー or BYOK が無いプロバイダは**ルーティング対象から自動除外**(plans/04 §8)。全プロバイダ未設定時は翻訳/チャットが可視的に失敗し原文フォールバック(P3)。

## 7. ホスティング手順書(今回の成果物)

`docs/deployment.md` を作成し、plans/01 §8 を運用手順に落とす:

- 単一オリジン Caddy(`/api/*`→FastAPI、他→Next.js)+ prod docker-compose(caddy/web/api/worker×2/postgres/redis/prometheus/grafana)。
- 環境変数チェックリスト(.env.production)、R2 設定、OAuth リダイレクト URI 登録、SMTP(SES)設定。
- `alembic upgrade head`(one-shot)→ イメージ push(GHCR)→ `docker compose pull && up -d` の順序。
- pg_dump 日次バックアップ / リストア手順。
- 拡張のストア申請手順(Chrome Web Store / Edge Add-ons、zip ビルド `pnpm --filter @alinea/extension zip[:edge]`、掲載文・スクショ・プライバシー記載)。

## 8. スコープと正直な見積もり

- 本実装は plans/13 の M0+M1+M2 に相当し、L 規模のサブシステムが数十個ある。クリティカルパス(取り込み・翻訳基盤→PDF→記事・図)を優先的に貫通させる。
- 各マイルストーンの検証ゲート(§3)を通過して初めて「そのマイルストーンは完璧に動く」と報告する。未達なら未達と明示し、実出力(テストログ)を添えて報告する。
- M3(他サイトアダプタ・OCR・Anki・PWA 等)は docs/10 §5 のとおり v1 スコープ外。
