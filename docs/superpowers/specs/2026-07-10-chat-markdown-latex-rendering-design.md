# チャット Markdown・LaTeX レンダリング設計

> 作成日: 2026-07-10 / 設計承認: ユーザー（2026-07-10）

## 1. 背景と目的

チャット API は、AI 回答に Markdown の見出し・表・コードブロックと、KaTeX 互換の
`$...$` / `$$...$$` を出力するようモデルへ指示している。一方、現在の Web 実装は
`[[ev:n]]`、`**太字**`、単一行の `$...$` だけを独自正規表現で処理し、回答全体を
`<p>` に入れている。そのため、Markdown 表やブロック数式がプレーンテキストまたは
インライン要素として扱われ、狭いチャット欄で重なり・はみ出し・行高不足が発生する。

本変更の目的は、AI 回答を標準的な Markdown と LaTeX として安全に描画し、既存の
根拠チップとストリーミング表示を維持したまま、チャット欄内のレイアウト崩れをなくす
ことである。

## 2. 対象範囲

### 対象

- アシスタントの `markdown` ブロック
- `outside_knowledge` / `speculation` の補足ブロック本文
- Markdown の見出し、段落、強調、順序付き・順序なしリスト、表、引用、水平線、リンク、
  インラインコード、フェンス付きコードブロック
- インライン数式 `$...$` とブロック数式 `$$...$$`
- Markdown 内の任意の通常テキスト位置にある `[[ev:n]]` 根拠マーカー
- SSE ストリーミング中の未完成 Markdown

### 対象外

- ユーザーメッセージ本文の Markdown 化（現状どおりプレーンテキスト）
- Markdown 内の生 HTML の実行
- Markdown 画像の読み込み・埋め込み
- Mermaid など、Markdown・KaTeX 以外の独自記法
- 記事モードの Markdown レンダラー変更
- API、DB、SSE イベント形式の変更

## 3. 採用方式

チャット専用の `ChatMarkdown` コンポーネントを追加し、標準 Markdown パイプラインを
使用する。

- `react-markdown`: Markdown AST から React 要素を生成する。
- `remark-gfm`: GitHub Flavored Markdown の表などを処理する。
- `remark-math`: `$...$` / `$$...$$` を数式ノードとして処理する。
- KaTeX: 数式を HTML に変換する。既存の KaTeX 設定・マクロと互換になるよう共通設定を
  再利用する。
- チャット専用 remark プラグイン: `[[ev:n]]` を根拠要素へ変換する。

既存の正規表現パーサーを拡張する方式は、表・入れ子・フェンス付きコード・未完成構文を
正しく扱い続けるコストが高いため採用しない。記事用の軽量パーサーとの共通化は、記事表示
への影響が今回の範囲を超えるため行わない。

## 4. コンポーネント構成とデータフロー

`ChatMessage` はメッセージ種別とアクション行の責務を維持し、本文描画だけを
`ChatMarkdown` へ委譲する。

1. `ChatMessage` がブロックの `text`、`evidence`、`onEvidenceJump` を
   `ChatMarkdown` へ渡す。
2. Markdown パーサーが GFM と数式を AST 化する。
3. 根拠プラグインが通常のテキストノードだけを走査し、`[[ev:n]]` を専用プロパティ付きの
   内部リンクノードに変換する。コード・インラインコード内の同じ文字列は変換しない。
4. カスタムリンクレンダラーは専用プロパティを持つノードだけを `EvidenceChip` にする。
   通常リンクは安全な外部リンクとして描画する。
5. 根拠番号が `evidence` に存在しない場合は、現在と同じくマーカーを表示しない。
6. 補足ブロックはラベルバッジを独立して表示し、その本文を同じ `ChatMarkdown` へ渡す。

これにより、段落・見出し・リスト・表セルのどこに根拠マーカーがあっても同じジャンプ動作を
維持する。ユーザー入力は `ChatMarkdown` を通さない。

## 5. Markdown と数式の表示規則

- 見出しはチャット内の階層を示しつつ、サイドパネルに収まる小さな文字サイズにする。
- Markdown 要素間には一貫した縦余白を設け、ルート先頭・末尾には余分な余白を作らない。
- 表は罫線とセル余白を持ち、外側コンテナを `overflow-x: auto` にする。列内容を潰して
  重ねず、必要時だけ横スクロールさせる。
- フェンス付きコードは等幅フォントと背景を持ち、改行を保持して横スクロール可能にする。
- 長い通常テキストとリンクはチャット幅で折り返す。
- `$...$` は文中のインライン数式として描画する。
- `$$...$$` は、区切り文字が同一行か別行かを問わずブロック数式として描画する。
- ブロック数式は独立した行と十分な上下余白を持ち、幅を超えた場合は数式部分だけを
  横スクロールさせる。行高不足による上付き・下付き・括弧の重なりを許さない。
- KaTeX が解釈できない入力ではコンポーネントを例外終了させず、元の LaTeX を読める
  フォールバック表示にする。

## 6. ストリーミング時の挙動

ストリーミング中も、受信済みの文字列全体を同じ `ChatMarkdown` で再解析する。閉じていない
強調、コードフェンス、表、数式区切りは、その時点でパーサーが解釈できる安全なテキスト
または部分構造として表示する。後続 delta で構文が閉じた時点で完成形へ切り替える。

未完成構文や不正な LaTeX によって React レンダリング、SSE 継続、メッセージアクションが
停止してはならない。既存の「生成中」表示、再生成、コピー、メモ保存、根拠ジャンプの挙動は
変更しない。

## 7. セキュリティ

- `rehype-raw` は使用せず、生 HTML は React 要素として実行しない。
- Markdown 画像は描画しないため、回答表示だけで外部画像へアクセスしない。
- 通常リンクは安全な URL だけを許可し、`target="_blank"` と
  `rel="noopener noreferrer"` を付ける。危険なスキームはリンク化しない。
- 根拠チップ判定はプラグインが付けた内部プロパティと、実在する `EvidenceRef` の両方を
  必須とする。AI が書いた通常リンクだけで根拠チップを偽装できないようにする。
- KaTeX は信頼モードを有効にせず、既存設定と同じくエラーを UI 全体へ送出しない。

## 8. テスト方針

既存の `ChatPanel.test.tsx` を維持しつつ、本文レンダラーを独立して検証する。

- 見出し、太字・斜体、順序付き・順序なしリスト、引用、水平線
- GFM テーブルの行・見出し・セルと横スクロール用ラッパー
- インラインコードとフェンス付きコード、およびコード内の根拠記法を変換しないこと
- 通常リンクの安全属性、危険 URL、生 HTML、画像の無効化
- `$...$` のインライン KaTeX と、同一行・複数行の `$$...$$` のブロック KaTeX
- 不正な LaTeX のフォールバック
- 段落・見出し・リスト・表セル内の `[[ev:n]]` が `EvidenceChip` になり、クリックで対応
  アンカーを渡すこと
- 未解決根拠マーカーを表示しないこと
- 補足ブロックでも同じ Markdown・数式レンダリングを使用すること
- ストリーミング途中の未閉じ Markdown / LaTeX が例外を起こさず、完成後に切り替わること
- ユーザーメッセージがプレーンテキストのままであること

変更後は対象 Vitest に加えて Web の型チェックと lint を実行する。

## 9. 受け入れ基準

- AI 回答の Markdown 表、見出し、リスト、コードブロックが構造化表示される。
- `$...$` と `$$...$$` がそれぞれインライン・ブロック数式として表示される。
- 添付例のような大きいブロック数式が上下に重ならず、チャット幅を超える場合だけ横へ
  スクロールできる。
- 表、コード、数式のいずれもチャットパネル全体の横幅を押し広げない。
- Markdown 内でも根拠チップの表示と本文ジャンプが維持される。
- 補足ボックスにも同じ Markdown・LaTeX 対応が適用される。
- 生 HTML、危険 URL、外部画像が回答表示から実行・取得されない。
- ストリーミング中の未完成構文で表示や受信が停止しない。
- 既存のチャット操作とユーザーメッセージ表示に回帰がない。
