# 最終ユーザー受け入れチェックリスト

このチェックリストは、論文プレゼンテーションおよびPPTX生成を除く実装を、リリース候補版でまとめて確認するための文書である。
開発途中でユーザー確認を行う場合も、該当する節だけを同じ手順で使用する。
全項目を一度に確認する場合の所要時間は2時間30分から4時間を見込む。
複数人で分担する場合も、最終判定者がFAIL、BLOCKED、N/Aの理由をまとめて確認する。
受け入れ結果は、この原本を作業用文書またはIssueへ複製して記録する。
リポジトリ内の原本を直接更新すると確認対象のコミットSHAが変わるため、リリース候補の作成後は原本を変更しない。

## 1. 判定方法

各項目は、操作と結果を確認してからチェックする。
画面を開いただけの場合や、実装者の説明だけを確認した場合はチェックしない。

- [ ] 確認対象のコミットSHAまたはビルド番号を記録した。
- [ ] 確認日時、ブラウザ、OS、端末を記録した。
- [ ] 本番の個人データではなく、受け入れ確認用のアカウントと論文を使用している。
- [ ] APIキー、認証情報、非公開論文の本文をスクリーンショットや不具合票へ記載しないことを確認した。
- [ ] `該当なし` とする項目には、理由を不具合記録へ残した。

判定は次の四種類から選ぶ。

- **PASS**：期待する結果を確認し、チェックを入れた。
- **FAIL**：期待する結果と異なる。
- **BLOCKED**：環境、権限、テストデータの不足により確認できない。
- **N/A**：今回の構成では対象外であり、理由を記録した。

次の問題が一件でも残る場合は、最終判定を `NO-GO` とする。

- P0：データ消失、他ユーザーへの情報漏れ、権限回避、秘密情報の露出、同意のない外部API課金。
- P1：主要な利用手順を完了できない、誤った結果を高い確度で表示する、復旧手段がない。
- BLOCKED：このチェックリストの必須項目を確認できない。

P2は回避策がある機能上の問題、P3は表示上の軽微な問題とする。
P2またはP3を残して公開する場合は、課題番号、担当者、対応予定を最終判定欄へ記録する。

## 3. 確認開始条件

実装者は、ユーザーへ確認を依頼する前に次の状態を用意する。

- [ ] API、Worker、Web、ブラウザE2Eの自動テストが成功している。
- [ ] DBマイグレーションの適用、巻き戻し、再適用が成功している。
- [ ] OpenAPIの生成結果に未反映の差分がない。
- [ ] 確認用の一般ユーザーA、一般ユーザーB、公開者ユーザーを用意した。
- [ ] ACL Anthology、OpenReview、PubMed Central、Hugging Face、arXivの確認用URLを一件以上ずつ用意した。
- [ ] 翻訳、用語、メモ、注釈、チャット、図表、関連ソースを含むバックアップ確認用論文を用意した。
- [ ] GitHubコード対応を人手で判断できる論文と公式リポジトリを用意した。
- [ ] 外部APIの利用額を確認できるテスト用の費用上限を設定した。
- [ ] 既知の制約と未解決課題の一覧を確認者へ渡した。

上記を満たさない項目は、機能不良と混同せず `BLOCKED` として記録する。

## 4. 対象範囲と既定値

- [ ] 論文プレゼンテーションおよびPPTX生成が今回の受け入れ対象外であることを確認した。
- [ ] GitHubコード対応解析の既定モードが `on_demand` である。
- [ ] GitHubコード対応解析の月額予算の既定値が5.00 USDである。
- [ ] Hugging Face由来のGitHub、プロジェクト、Model、Dataset、Spaceは、利用者が採用するまでは候補として表示される。
- [ ] 外部サイトから取り込んだ論文は、共有可能なライセンスを機械判定できない限りprivateになる。
- [ ] 公開記事に原文、訳文、メモ、チャット、非公開の議論が含まれない。

## 5. バックアップと既存データの保全

一般ユーザーAのデータをエクスポートし、一般ユーザーBへ二回インポートする。
ユーザーBには、インポート前から別の設定、メモ、用語を保存しておく。

- [ ] エクスポート画面が、実際に含まれるデータ種別を正しく表示する。
- [ ] 論文の書誌情報、要約、カテゴリ、ライセンス、公式リポジトリURLが復元される。
- [ ] 共有翻訳とユーザー専用翻訳が復元され、両者の区別が維持される。
- [ ] 論文用語集とユーザー用語集が復元される。
- [ ] メモ、注釈、チャット、既読位置、優先度、タグが復元される。
- [ ] 図、表、数式画像、サムネイルが欠けずに表示される。
- [ ] 関連ソースの状態が、確定、候補、却下の区別を保って復元される。
- [ ] ユーザーBがインポート前から持っていた設定、メモ、用語が上書きされない。
- [ ] 同じバックアップを二回取り込んでも、論文、翻訳、用語、関連ソースが重複しない。
- [ ] 壊したZIPまたは検証値の異なるファイルを取り込むと、何が不正かを示して安全に停止する。
- [ ] 失敗したインポートの途中データが残らない。

この節で欠落、上書き、重複、参照切れが一件でもあればP0とする。

## 6. 閲覧画面の生成機能

### 再翻訳

- [ ] 一つの段落を選び、追加指示を入力して再翻訳を開始できる。
- [ ] 再翻訳中も現在の訳文が消えない。
- [ ] 新しい訳文を採用すると、その段落だけが更新される。
- [ ] 新しい訳文を破棄すると、現在の訳文が変わらない。
- [ ] 失敗時に再実行でき、空の訳文で上書きされない。

### チャットと根拠

- [ ] 論文について質問し、回答を生成できる。
- [ ] 回答の根拠となる論文箇所が表示される。
- [ ] 根拠を選ぶと、現在の版の該当箇所へ移動できる。
- [ ] 根拠が取得できない場合に、存在しない引用箇所を作らない。
- [ ] 生成に失敗した場合も質問文が失われず、再実行できる。

### AI単語候補

- [ ] 論文から単語候補を生成できる。
- [ ] 候補の語、意味、根拠となる箇所を確認できる。
- [ ] 一件を採用すると用語集へ追加され、再読込後も残る。
- [ ] 一件を却下すると候補一覧から外れ、再生成で直ちに復活しない。
- [ ] 候補がない場合と生成に失敗した場合を区別して表示する。

### arXiv改版差分

- [ ] 二つの版を選択して差分を生成できる。
- [ ] 追加、削除、変更なしを見分けられる。
- [ ] 差分から該当する論文箇所へ移動できる。
- [ ] 同じ版同士を選んだ場合に、誤った差分を表示しない。
- [ ] 差分生成の失敗理由と再実行手段が表示される。

## 7. 論文単位のスタンドアロン出力

- [ ] 原文、訳文、対訳、記事、注釈PDFから必要な成果物だけを選べる。
- [ ] 生成できない成果物は、選択可能であるかのように表示されない。
- [ ] 生成中、完了、失敗を画面から判別できる。
- [ ] 失敗後に同じ条件で再実行できる。
- [ ] ZIPを別端末へ移し、ネットワークを切った状態でHTMLを開ける。
- [ ] オフラインHTMLで本文、数式、図、表、内部リンクを表示できる。
- [ ] 原文HTMLの不要な抽出記号や壊れたタグがWeb版と同じ基準で除去される。
- [ ] 注釈PDFの注釈が対応する原文箇所に付いている。
- [ ] 対訳PDFが原文ページ、訳文ページの順で交互に並ぶ。
- [ ] 選択しなかった成果物がZIPへ含まれない。
- [ ] Anki TSVに改行またはタブを含む値があっても列がずれない。

## 8. ユーザー別LLM設定

一般ユーザーAと一般ユーザーBで、異なるprovider、model、BYOKを設定して同時に処理を開始する。

- [ ] 各ユーザーの処理が、そのユーザーが選んだproviderとmodelを使う。
- [ ] 翻訳、要約、用語抽出、チャット、埋め込み、GitHubコード解析で設定が一致する。
- [ ] 一方のユーザーのBYOKが、他方の処理、画面、ログへ現れない。
- [ ] BYOKがジョブpayload、バックアップ、エラーメッセージへ現れない。
- [ ] 利用できないproviderを選んだ場合に、設定した方針どおり停止またはフォールバックする。
- [ ] 実行履歴から使用したproviderとmodelを確認できる。

設定の混在または秘密情報の露出はP0とする。

## 9. 外部サイトとHugging Face

ACL Anthology、OpenReview、PubMed Central、Hugging Faceの確認用URLを順に取り込む。

- [ ] 対応URLを貼り付けると、対応サイトとして判定される。
- [ ] タイトル、著者、公開日、識別子が取得できる。
- [ ] 取得可能な本文またはPDFが通常の論文処理へ渡る。
- [ ] 同じ論文を別URLから取り込んでも、重複する論文が作られない。
- [ ] 認証が必要なページや取得できない本文を、取得成功として扱わない。
- [ ] 外部サイトの障害時に、失敗理由と再試行手段が表示される。
- [ ] Hugging FaceページからarXivリンクを見つけ、既存論文または新規取り込みへ結び付ける。
- [ ] Hugging FaceページからGitHubとプロジェクトページを関連ソース候補として表示する。
- [ ] 関連するModel、Dataset、Spaceを候補として表示する。
- [ ] 候補を採用すると確定した関連ソースになり、却下すると再読込後も却下状態を保つ。
- [ ] 候補URLの追跡パラメータや表記差を正規化し、同じリンクを重複表示しない。
- [ ] Hugging Face上に関連リンクがない場合に、存在しないリンクを推測して作らない。
- [ ] 利用者が設定した公式リポジトリURLを、自動収集した候補が上書きしない。
- [ ] 本文または必須アセットが欠けている間は、論文を利用可能な状態として表示しない。
- [ ] 処理完了と表示された論文は、閲覧画面と選択可能な出力を実際に利用できる。

## 10. セマンティック検索と似た論文

事前に、検索結果の正解が分かる五つの質問と対象論文を用意する。
日本語、英語、論文中に同じ単語がない言い換えを含める。

- [ ] 既存のキーワード検索が従来どおり利用できる。
- [ ] 日本語の意味検索で意図した論文を見つけられる。
- [ ] 英語の意味検索で意図した論文を見つけられる。
- [ ] 表記が異なる言い換えでも関連する論文を見つけられる。
- [ ] 五つの確認質問のうち四つ以上で、期待する論文が上位五件へ入る。
- [ ] 論文画面から「似た論文」を表示できる。
- [ ] 一般ユーザーAの検索結果に、一般ユーザーBだけが持つprivate論文が現れない。
- [ ] 埋め込みproviderを失敗させても、キーワード検索結果だけで応答する。
- [ ] 空の検索結果と検索障害を区別して表示する。

他ユーザーのprivate論文が検索結果へ現れた場合はP0とする。

## 11. GitHubコード対応解析

論文中の主張と対応コードを人手で確認済みのサンプルを使う。
少なくとも五つの主張と、対応しない主張を一つ含める。

### 実行モードと費用

- [ ] `off` では解析ボタンまたは自動処理から外部API呼び出しが発生しない。
- [ ] `on_demand` では、利用者がボタンを押すまで解析が始まらない。
- [ ] 実行前に対象リポジトリ、見積額、残予算を確認できる。
- [ ] 確認操作の後にだけ外部API呼び出しが始まる。
- [ ] `automatic` では、採用済みのGitHubソースまたは高確度の公式候補がある場合だけ開始する。
- [ ] 弱い候補または却下済み候補だけでは自動解析を開始しない。
- [ ] 月額予算を超える場合は外部APIを呼ばず、予算待ちであることを表示する。
- [ ] 使用額と残予算が実行後に更新される。

### 解析結果

- [ ] 論文の主張から対応するファイル、symbol、行範囲を確認できる。
- [ ] GitHubリンクがブランチの先端ではなく固定commitを指す。
- [ ] リンク先の行が、画面に示された主張と実際に対応する。
- [ ] 人手で用意した五つの対応のうち四つ以上を妥当に説明できる。
- [ ] 対応しない主張を、根拠のないコードへ結び付けない。
- [ ] 確度が低い結果は、その状態を明示するか表示対象から外す。
- [ ] private、404、巨大すぎる、対応外形式のリポジトリを安全に拒否する。
- [ ] リポジトリのコードやインストールスクリプトを実行しない。
- [ ] APIキー、秘密ファイル、リポジトリ外のファイルを解析対象へ含めない。

誤った対応を高い確度で表示する場合はP1とする。
利用者の確認なしに予算を超えてAPIを呼ぶ場合はP0とする。

## 12. オフライン閲覧

- [ ] オンラインで訪問した論文を、ネットワークを切った後に再度開ける。
- [ ] オフラインでも本文、訳文、数式、図、表を表示できる。
- [ ] 訪問していない論文をオフラインで開くと、未保存であることを表示する。
- [ ] 最近訪問した論文が上限に従って保存され、古い論文から削除される。
- [ ] 一論文50 MiBおよび全体200 MiBの上限を超えた場合に安全に縮退する。
- [ ] オンライン時の401応答がキャッシュで隠されず、ログイン画面へ進む。
- [ ] ログアウト後に前のユーザーの論文をオフライン表示できない。
- [ ] 同じ端末で別ユーザーへ切り替えても、前のユーザーのキャッシュが表示されない。

ログアウト後またはユーザー切替後にprivate論文を表示できる場合はP0とする。

## 13. 記事公開とコメント

private論文から、限定公開記事と公開記事を一件ずつ作る。
ログアウト状態、別ユーザー、公開者の三つの状態で確認する。

- [ ] 限定公開記事はURLを知る利用者だけが閲覧できる。
- [ ] 限定公開記事に `noindex` が設定される。
- [ ] 公開記事をログアウト状態で閲覧できる。
- [ ] 公開記事に原文、訳文、メモ、注釈、チャット、privateな関連ソースが含まれない。
- [ ] 公開後に元の論文を編集しても、公開済みスナップショットが意図せず変わらない。
- [ ] 匿名利用者は記事を閲覧できるが、コメントを投稿できない。
- [ ] 認証済み利用者はコメントを投稿できる。
- [ ] コメント投稿者は自分のコメントを編集および削除できる。
- [ ] 別ユーザーは他人のコメントを編集または削除できない。
- [ ] 公開者はコメントを非表示にし、必要なら再表示できる。
- [ ] 投稿数の上限を超えると、理由と再試行可能な時期を表示する。
- [ ] 公開停止後に、記事とコメントが公開画面へ残らない。

privateな内容の混入または権限外操作の成功はP0とする。

## 14. 通常利用の通し確認

新しいユーザーを作り、Hugging FaceまたはarXivのURLから一連の操作を行う。

- [ ] URLを取り込み、処理完了まで状態を確認できる。
- [ ] 論文を読み、翻訳、検索、メモ、注釈を利用できる。
- [ ] Hugging Faceまたは本文から見つかった関連ソースを採用できる。
- [ ] 必要な場合だけGitHubコード対応解析を開始できる。
- [ ] 論文単位の成果物を生成してダウンロードできる。
- [ ] オフラインで訪問済み論文を再読できる。
- [ ] 私的データを含めずに記事を公開できる。
- [ ] 途中でブラウザを再読込しても、完了済みの状態と進行中ジョブを追跡できる。
- [ ] 主要な操作をモバイル幅でも完了できる。
- [ ] 利用者が回復できないエラー、無反応なボタン、終了しない進行表示がない。
