# 仕様ドキュメント

> 要旨: 論文読解ワークベンチ「訳読 / YAKUDOKU」の詳細仕様の目次と読み方。仕様は確定デザイン(画面ID 1a〜5a)を正とし、全13文書(00〜12)で構成する。[00-product.md](00-product.md) のプロダクト原則と主要シナリオが全体の判断基準であり、各機能仕様はそこから逆算して書かれている。

## 1. 対象プロダクト

**訳読 / YAKUDOKU — 論文読解ワークベンチ**。英語論文を日本語で深く読むための研究者向けツールで、対訳を軸に、翻訳・注釈・読解チャット・語彙学習・外部リソース参照までを1つのワークベンチにまとめる。通し例の題材は実在論文 Rectified Flow (arXiv:2209.03003)。

## 3. 読み方

- **はじめて読む**: 00 → 10(何を・どの順で作るか)→ 興味のある機能へ。ビューア中心の機能は 04 → 05 → 07 → 11 → 12 の順が読みやすい。
- **実装を始める**: 01(ドメインモデル)と 09 §5(ライセンス制約)は全機能の前提。UI 実装はさらに 09 のデザイントークン(アクセント4色・注釈4色・ステータス6色・書体4系統、基準ビューポート 1440×900、ダークモード)を先に確認する。
- **正とする情報源**: 確定デザイン(画面抽出)と docs が矛盾する場合はデザインが正。各文書はデザインの画面IDを (1a)(4d) のような形式で参照する。

## 4. 表記

- 各文書の冒頭に「> 要旨」、末尾に受け入れ基準チェックリスト(`- [ ]`)を置く。
- 各文書内の「決定」には理由を併記している。未決事項は 00 の Q1〜Q5 に集約。
- UI 文言はデザインの逐語を「」で引用する(例: 品質バッジの説明文「LaTeX ソースから完全構造化。数式・相互参照・図表・脚注を保持しています。」)。
- 品質レベルは **A(LaTeXから完全構造化)/ B(PDF由来)** の2段階。旧 A〜D 4段階の記述は無効。
- 文書間の参照はファイルリンク(例: [04-viewer.md](04-viewer.md))を基本とし、他文書の節番号への深い参照は避ける。

## 5. 受け入れ基準

- [ ] 目次が 00〜12 の全13文書を網羅し、各説明が当該文書の実内容と一致している

- [ ] 画面ID(1a〜5a)から担当文書に迷わず辿り着ける
