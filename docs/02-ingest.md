# 02. 論文の取り込み

> 取り込みは体験の入口。原則 P2(1アクション・段階的提示)と P3(黙って壊れない)を最も強く適用する。確定デザインに合わせ、**取り込み経路はブラウザ拡張(3a)に一本化**し、品質レベルは **A / B の2段階** とする。

## 1. 入力経路

取り込みはブラウザ拡張(Chrome / Edge)が唯一の経路。デザイナー注記のとおり「ツールバー → 保存の2クリックでライブラリへ / URLのみ送信・解析はサーバー」(3a)。アプリ内の「+追加」ボタン/取り込みダイアログ、および D&D による PDF アップロードは**廃止**する。

- 理由: 論文と出会う場所はブラウザ(arXiv・検索結果・出版社ページ)であり、その場で保存できる拡張が最短経路。アプリ側に別の取り込み UI を持つと導線が二重化し、書誌の自動抽出・保存済み検出などの体験が分岐してしまう。

| 経路 | 入力 | 備考 |
|---|---|---|
| 拡張: arXiv URL 送信 | 現在タブの arXiv abs ページ URL | 「arXiv 論文を検出」バッジ。**URL のみをサーバーへ送信し、取得・解析はサーバーで実行**(ページ内容は送らない)。書誌は保存前に自動抽出してプレビュー |
| 拡張: タブ内 PDF 直接送信 | 現在タブで表示中の PDF バイト列 | **明示操作のみ**(「このタブの PDF を送信」ボタン。自動送信はしない)。サーバーから取得できない学内ネットワーク等を想定。「private 論文として保存され、共有されません」。書誌はローカル推定(「書誌は推定」バッジ) |
| 対応サイトアダプタ | OpenReview / ACL Anthology / PubMed / PMC / Hugging Face の URL | サイト別アダプタで検出対象を広げる。Hugging FaceはarXiv同定と関連ソース収集に使う。§8 |

拡張ポップアップの UI 詳細・4状態(保存前/保存直後/既存/一般 PDF)は [08-extension.md](08-extension.md)。

### 1.1 保存前プレビューと保存時入力(3a 保存前状態)

保存ボタンを押す前に、ポップアップ内で以下が確認・指定できる。

| 項目 | 内容 | 既定 |
|---|---|---|
| 書誌プレビュー | タイトル・著者短縮表記・会議・arXiv ID+バージョン(例: 「Liu, Gong, Liu · ICLR 2023 · arXiv:2209.03003 v3」) | 自動抽出 |
| 品質見込み | LaTeX ソース有無を**保存前に判定**し「✓ LaTeX ソースあり — 品質レベル A 見込み」と予告(§4) | — |
| ステータス | 3択の単一選択ピル: 「読む予定」「すぐ読む」「読んでいる」 | 「読む予定」 |
| タグ | チップ削除(×)・自由追加(「追加…」)・提案タグのワンクリック追加(「提案: distillation +」) | arXiv カテゴリ等から提案(§7) |
| 追加先 | コレクション選択ドロップダウン(例: 「輪読会 2026-07 ▾」) | なし |
| ひとことメモ | 短文メモ入力(「ひとことメモ…」) | 空 |

- 保存はボタンクリックまたは **Enter キー**(「保存 ⏎」)。指定しなければ既定値で即時実行し、あとから変更できる(管理を強要しない)。
- プライバシー注記を常時表示: 「URL のみを送信します — 取得・解析はサーバーで実行」。

## 2. arXiv の解決フロー

1. URL / ID を正規化する(`arxiv.org/abs/2209.03003v3`, `/pdf/...`, `/html/...`, 旧形式 `cs/9901002` をすべて `arxiv_id + version` に解決)。
2. メタデータ API で書誌・アブストラクト・カテゴリ・**ライセンス** を取得する(ライセンスは図表転載可否の判定に使う。例: 「CC BY 4.0 — 図表転載可」(2a))。
3. ソースを以下の優先順位で取得し、最初に成功した形式でパースする:
   **LaTeX ソース > arXiv HTML > PDF**。
   - 理由: 確定デザインの取り込みタイムライン(2a)は「arXiv から LaTeX ソース取得」を第1段に置いており、LaTeX が主経路。LaTeX は数式・相互参照・図表・脚注の完全な構造を持つ。arXiv HTML(公式 HTML / ar5iv)は LaTeX から機械生成されるためフォールバックとして同等の構造化が可能。PDF は最終フォールバック。
   - **決定**: arXiv HTML 経由で完全構造化できた場合も品質レベル A とする。理由: A の定義は「LaTeX ソースから完全構造化」であり、arXiv HTML は LaTeX ソース由来の生成物で保持情報が同等のため。どの形式で取り込めたかは処理ログ(§5.3)に記録し、ユーザーが確認できる。
   - **段階導入**: 上記の優先順位は最終状態。M0 では実装コストの低い arXiv HTML を主経路とし、M2 で LaTeX ソース解析を主経路に昇格する([10-roadmap.md](10-roadmap.md))。
4. 拡張は保存前に LaTeX ソースの有無をサーバーへ照会し、「品質レベル A 見込み」を予告する(§1.1)。arXiv abs ページの「TeX Source」リンク相当の存在確認で判定する。
5. バージョン指定なしの URL は最新バージョンを取る。既にライブラリに旧バージョンがある場合は §6 の統合フローへ。

## 3. PDF パイプライン(品質レベル B)

タブ内 PDF 送信、および arXiv でソース・HTML が取得できず PDF にフォールバックした場合のパイプライン。ベストエフォートであることを UI 上も明示する(ソースバッジ「B」+「PDF 取り込み」表示)。

- 段組み解析(1/2 カラム判定)、読み順復元、ヘッダ・フッタ・ページ番号の除去
- 見出し検出(フォントサイズ・番号パターン)→ セクションツリー構築
- 図・表領域の検出とキャプションの対応付け。図はページから画像として切り出す
- 数式領域の検出: 認識できれば LaTeX 化、できなければ画像として埋め込み(どちらも品質レベルは B のまま。到達度は処理ログに記録)
- 参考文献セクションの検出とエントリ分割
- 各ブロックに `page + bbox` を保持し、PDF モードとの相互リンク([04-viewer.md](04-viewer.md))に使う
- メタデータ: PDF 内埋め込み情報 + 1ページ目からのタイトル・著者・アブスト推定 + DOI 文字列検出 → 外部書誌 API で補完。推定メタデータには「書誌は推定」バッジを付け、**編集可能なフォームで確認できる**(誤推定を直せないと検索性が死ぬ)
- テキストレイヤが無い、または可視テキストが不十分な PDF は、通常解析の後に **Tesseract OCR を最終候補**として試す。OCR 結果にも段落数・本文被覆率・図表資産の完全性を同じ基準で適用し、不十分な結果は採用しない(P3: 黙って壊れない)。ページ全面のスキャン画像は背景として除外し、独立した図表だけを表示資産として保持する。
- OCR は Linux 上の分離子プロセスで実行し、時間・メモリ・ページ数・テキスト量・図表数／バイト数を制限する。Tesseract 本体または言語データが無い場合は通常 PDF を止めず、OCR が必要なジョブだけ理由付きで終了する。

## 4. 品質レベル(A / B の2段階)

取り込み結果に必ず品質レベルを付与し、ライブラリカード・ビューアヘッダ・情報パネルにバッジで**常時明示**する(P3)。旧 A〜D の4段階は廃止し、旧 C(テキストのみ)/ D(OCR)は B に統合する。OCR の使用有無・言語・失敗履歴は処理ログと revision stats に保持する。

| レベル | 意味 | 由来 | 体験 |
|---|---|---|---|
| **A** | 完全構造化 | LaTeX ソース(または arXiv HTML) | 「LaTeX ソースから完全構造化。数式・相互参照・図表・脚注を保持しています。」(2a 情報パネルの定義文言)。全機能が最良動作 |
| **B** | PDF 由来 | PDF 解析 | ベストエフォートの構造化。段落・見出し・図・キャプションを可能な範囲で復元。数式は LaTeX or 画像。ライブラリでは「B PDF 取り込み」と表示(1d) |

- バッジ表示箇所: ビューアヘッダの「A」バッジ、情報パネル「品質レベルと取り込み」セクション(2a)、ライブラリカード・テーブルのソースバッジ「A」(アクセント淡色)/「B」(グレー)(1d)。
- B の論文では、構造化の到達度(数式 LaTeX 化率・参照リンク等)を処理ログ(§5.3)から確認できる。
- **昇格(B → A)**: PDF 取り込み後に arXiv 版(LaTeX ソース)が見つかった場合、再パースして A へ昇格できる。昇格時は新 DocumentRevision を作り、注釈をリアンカーする([01-domain-model.md](01-domain-model.md))。昇格は自動適用せず**アプリ内通知で提案し、ワンクリック適用**(通知 UI は [06-library.md](06-library.md)。P6: 勝手に状態を変えない)。

## 5. 取り込みパイプラインと段階的提示

### 5.1 ステートマシン

`queued → fetching → parsing → structuring → translating_abstract → readable → translating_body(バックグラウンド) → complete`(任意の段階から `failed(stage, reason)`)。

内部状態はユーザーには3段階に集約して見せる。表示面ごとの対応:

| 表示面 | 表示(逐語) |
|---|---|
| 拡張ポップアップ・保存直後(3a) | 「✓ 書誌」→「✓ 構造化」→「翻訳中 12%」+プログレスバー。「進捗はツールバーのバッジにも表示されます(処理中スピナー → 完了チェック)。読める部分から開けます。」 |
| ダッシュボード「最近追加」カード(1d) | 「✓ 書誌」→「✓ アブスト訳・要約」→「本文翻訳中 68%」+進捗バー。「§3 まで読めます · 開いたセクションを優先翻訳」+「読み始める」ボタン |
| 情報パネル・取り込みタイムライン(2a) | §5.3 の3段タイムライン(完了後の記録) |

### 5.2 段階的提示(P2)の時間目標

| 時点 | 目標 |
|---|---|
| カードがライブラリに現れる(書誌+サムネイル仮) | 保存後 3 秒以内 |
| アブストの日本語訳+「✦ 3行要約」(①②③形式)がカードに出る | 20 秒以内 |
| 本文を読み始められる(readable: 最初のセクション訳済み) | 60 秒以内 |
| 標準的な論文(〜10ページ)の全文翻訳完了 | 5 分以内 |

- `readable` 到達時点でビューアを開ける(**部分読書**)。カードには「§3 まで読めます」のように読書可能範囲を示し、「読み始める」で翻訳完了前でも読書を開始できる。未翻訳セクションは原文表示+翻訳中インジケータで、完了分から順に差し替わる。
- ユーザーが開いているセクションを翻訳キューの先頭に繰り上げる(「開いたセクションを優先翻訳」= オンデマンド先行)。
- 付録は既定で自動翻訳しない(設定 4f)。未翻訳の付録は目次に「開くと翻訳します(オンデマンド)」と表示する([03-translation.md](03-translation.md))。
- 取り込み中のカードには段階名と進捗バーを表示する。失敗時は段階・理由・再試行ボタン(P3)。
- 全文翻訳完了はアプリ内通知(「読み始める →」アクション付き)+拡張のツールバーバッジで知らせる([06-library.md](06-library.md))。

### 5.3 取り込みタイムラインと処理ログ(2a 情報パネル)

取り込み完了後、ビューア情報パネルの「品質レベルと取り込み」セクションに**3段のタイムライン**を各段のタイムスタンプ付きで記録・表示する。

1. 「7/02 21:04 — arXiv から LaTeX ソース取得」(ソース取得。B の場合は PDF 取得)
2. 「21:05 — 構造化・図表抽出(24p / 図8 / 表4)」(ページ数・図数・表数の結果メタを併記)
3. 「21:09 — 全文翻訳 完了(自然訳 · v3)· 付録は未翻訳」(翻訳スタイル・版・未翻訳範囲を併記)

- タイムライン直下にアクションリンク「再取り込み」「処理ログ」を置く。
  - **再取り込み**: ソース再取得からパイプラインを再実行する(パーサ改良後の品質改善、B→A 昇格の手動実行を含む)。再取り込みも新 DocumentRevision + 注釈リアンカーの手順に従う。
  - **処理ログ**: 各段階の詳細ログ(取得形式、フォールバック発生、構造化の到達度、失敗理由)を表示する。「黙って壊れない」の受け皿。

## 6. 重複検知と統合

判定順: ① arXiv ID(バージョン無視)→ ② DOI → ③ PDF SHA-256 → ④ タイトル正規化+第一著者姓+年のファジー一致(閾値超えは「同一の可能性」として確認 UI)。

- **完全一致**: 新規作成せず、拡張ポップアップが「既にライブラリにあります」状態(3a 状態3)になる。**重複保存 UI は出さない**。現在ステータスピル・追加日・読書進捗(「2026/07/02 追加 · 進捗 42%」)・前回位置(「前回: §2.1 整流フロー · 昨日 21:52」)を表示し、「続きから開く ↗」「ステータス変更 ▾」を提供する。
- **別バージョン**(v1 所持中に v2 を追加): 同一 Paper の新 `source_version` として取り込み、「新しいバージョンがあります」バナーをビューアに表示。切り替えはユーザー操作(自動置換しない — 注釈の安全のため)。旧版注釈はリアンカーして引き継ぐ。バージョン間差分表示は**実装済み**(block 単位 diff エンジン+ `viewer_revision_diff` API、[10-roadmap.md](10-roadmap.md))。
- **ファジー一致のみ**: 自動統合しない。取り込み完了後に「この論文と同じですか?」カードを表示し、ユーザーが統合を確定する。
- タブ内 PDF 送信(private)で取り込んだ論文が後に arXiv 経路で一致検出された場合も同一 Paper に統合し、§4 の昇格フローに乗せる。

## 7. サムネイルと初期タグ

- サムネイル選択: ①論文中の Figure 1、②なければ最大面積の図、③図が無ければ概要図生成後に差し替え([07-figures-and-articles.md](07-figures-and-articles.md))、④それも無ければタイトルカード。ユーザーは図表一覧から手動で差し替え可能。
- 生成前のカードは破線プレースホルダ(「…」)で未生成を表す(1d)。
- タグ提案は2箇所で行い、いずれも**承認式**(自動付与はしない。タグ体系はユーザーの所有物):
  - 保存前: 拡張ポップアップの「提案: distillation +」(§1.1)。
  - 取り込み後: ライブラリカードの提案タグチップ「提案: cs.CV +」「solver +」— クリックで確定タグに昇格。
- 提案の材料: arXiv カテゴリ(cs.CV 等)、ライブラリ内の共起タグ、アブストからの抽出。

## 8. 対応サイト(アダプタ方針)

サイトごとに「検出+メタデータ取得+本文取得」のアダプタを追加し、拡張の自動検出対象を広げる構成。検出優先順は ACL Anthology → OpenReview(ICLR/NeurIPS レビュー付き)→ PubMed/PMC → Hugging Face関連ソース(`alinea_core.adapters.registry`)。**ACL Anthology / OpenReview / PubMed / PMC / Hugging Face は実装済み**。出版社ページ(IEEE/ACM)は追加候補として残る。

- Hugging Face Paper URLはarXiv IDを解決し、本文取得と翻訳は既存arXivパイプラインへ委譲する。
- Hugging Face Paper APIのGitHub、project、Model、Dataset、Spaceは本文候補ではなくResources候補へ保存する([12-resources.md](12-resources.md))。
- 採用したGitHub実装は、ユーザー設定に応じて論文blockとcode file／symbol／行範囲の対応解析へ渡せる。

- 認証が必要なページ・サーバーから取得できないページは、現行どおり拡張の**タブ内 PDF 直接送信**(明示操作・private)で受ける(§1)。
- 出版社コンテンツは private 扱いとし共有機能を無効化する([09-nonfunctional.md](09-nonfunctional.md))。

## 9. 受け入れ基準

- [ ] 取り込みはブラウザ拡張からのみ実行でき、アプリ内に「+追加」ボタン・取り込みダイアログ・D&D アップロードが存在しない
- [ ] arXiv abs ページで拡張を開くと、保存前に書誌プレビューと LaTeX ソース判定(「品質レベル A 見込み」)が表示される
- [ ] 保存時にステータス(3択・既定「読む予定」)・タグ(提案含む)・コレクション・ひとことメモを指定でき、Enter キーで保存できる
- [ ] arXiv 経路では URL のみが送信され、タブ内 PDF はボタン明示操作時のみ送信される(自動送信が発生しない)
- [ ] タブ内 PDF から取り込んだ論文が private として保存され、共有対象にならない
- [ ] LaTeX ソースが取得できる論文が品質レベル A で取り込まれ、A の定義文言と取り込みタイムライン3段(ソース取得→構造化・図表抽出→全文翻訳)がタイムスタンプ付きで情報パネルに表示される
- [ ] 品質レベルが全論文に A / B バッジで表示され、B では「PDF 取り込み」由来であることが読む前に分かる
- [ ] 保存後 3 秒以内にカード表示、20 秒以内に「✦ 3行要約」、60 秒以内に読み始められる(p50)
- [ ] 翻訳完了前のカードにパイプライン進捗と読書可能範囲(「§n まで読めます」)が表示され、「読み始める」で部分読書できる
- [ ] 同一論文を 2 回保存しても LibraryItem が重複せず、拡張に既存の進捗・前回位置と「続きから開く」が表示される
- [ ] 「再取り込み」「処理ログ」が情報パネルから実行・閲覧でき、取り込み失敗時は失敗段階・理由・再試行手段が UI に表示される
- [ ] B の論文に LaTeX ソースが見つかった場合、A への昇格が通知で提案され、ワンクリックで適用できる(自動適用しない)
- [ ] ACL Anthology / OpenReview / PubMed / PMC / Hugging Face の URL が検出優先順どおりにアダプタで取り込める(PMC OA 記事は JATS で品質 A を優先)
- [ ] 同一 Paper の別バージョン取り込み時にバージョン間差分を表示できる(`viewer_revision_diff`)
