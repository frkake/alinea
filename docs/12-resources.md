# 12. リソース(外部資料の添付)

> 論文外の学習材料 — 実装コード・発表動画・スライド・解説記事 — を論文にひも付ける機能。「論文を深く理解する材料は論文の外にもある」を1つのワークベンチに収める。URL を貼り付けるだけで種類を自動判定し(P2)、arXiv 由来の公式実装は自動検出して提案する(勝手に追加はしない。P6)。確定デザイン (5a) を正とする。

## 1. 目的と位置づけ

- 論文を読むとき、実装コード・著者の発表動画・スライド・第三者の解説記事を並行して参照することが多い。これらをブラウザのブックマークに散らさず、**論文単位でひも付けて 1 クリックで開ける** ようにする。
- ビューアのサイドパネル(排他 6 タブ: チャット / メモ / 注釈 / 図表 / **リソース** / 情報)の 1 タブとして提供する([04-viewer.md](04-viewer.md))。タブには件数バッジ(例: 「リソース 4」)が付く。
- リソースは閲覧の入口であり、コンテンツ本体は取り込まない(外部リンク+メタデータ+ひとことメモのみ)。著作権上も安全側(P7)。

## 2. リソースの種類と URL 自動判定

リソース追加は **URL 貼り付けのみ**(フッター入力、§6)。種類はユーザーに選ばせず URL から自動判定する。フッターのプレースホルダは「URL を貼り付け — 種類を自動判定」。

| 種類(kind) | 表示アイコン | 例(デザイン掲載) |
|---|---|---|
| `github`(GitHub 実装) | 「GH」黒地バッジ | gnobitab/RectifiedFlow |
| `youtube`(YouTube 動画) | 「▶」赤地バッジ | ICLR 2023 Oral — Flow Straight and Fast |
| `slides`(発表スライド・PDF) | 「PDF」アンバー地バッジ | 発表スライド(ICLR 2023) |
| `article`(解説記事) | サービス頭文字バッジ(例: Zenn=「Z」) | Rectified Flow を図で理解する |
| `huggingface`(Paper / Model / Dataset / Space) | 「HF」黄地バッジ | huggingface.co/papers/2307.09288 |
| `project`(公式プロジェクトページ) | 「PJ」青地バッジ | 研究プロジェクトのデモ・解説ページ |

**決定: 判定規則(ドメイン・パスパターン)** — デザインは「自動判定」の事実のみを示すため、規則を以下に確定する。上から順に評価する。

1. `github.com/{owner}/{repo}`(および配下パス)→ `github`。リポジトリ名は `owner/repo` に正規化する。Gist・GitHub Pages は該当しない(→ 4 へ)。
2. `huggingface.co/papers/{arxiv_id}`、`huggingface.co/{owner}/{repo}`、`huggingface.co/datasets/{owner}/{repo}`、`huggingface.co/spaces/{owner}/{repo}` → `huggingface`。
3. `youtube.com/watch?v=…` / `youtu.be/…` / `youtube.com/live/…` → `youtube`。
4. パスが `.pdf` で終わる URL、または取得時の `Content-Type: application/pdf`、または Speaker Deck / SlideShare のドメイン → `slides`。
5. Hugging Face Paper APIの `projectPage` から得たURL → `project`。
6. **上記いずれにも該当しない場合は `article` を既定とする**。理由: 解説記事のドメインは無数にあり列挙不能。誤判定しても表示差はアイコンとメタ項目のみで、リンクとしての機能は損なわれない(黙って壊れない。P3)。

- **決定: 種類はあとから変更できる**(カードメニュー、§7)。自動判定の誤りをユーザーが 1 操作で正せるようにするため。
- 同一 URL の重複追加は追加時に検知し、既存カードをハイライトして知らせる(重複登録しない)。**決定**: URL は正規化(トラッキングパラメータ除去等)して比較する。

## 3. メタデータの自動取得

追加直後にサーバー側で取得する(フッターのヘルプ文言: 「GitHub・YouTube・スライド・解説記事など。タイトルとサムネイルは自動取得、ひとことメモを添えられます。」)。

### 3.1 共通メタ

| 項目 | 内容 |
|---|---|
| タイトル | ページタイトル/リポジトリ名/動画タイトル |
| サムネイル | YouTube は必須表示(カード内 96px+右下に再生時間バッジ「12:34」)。他種類は取得しても v1 では表示しない(デザインどおりコンパクト表示) |
| ソースドメイン/サービス名 | メタ行の先頭に表示(「GitHub」「YouTube」「iclr.cc」「zenn.dev」) |

### 3.2 種類別メタ(カードのメタ行、逐語例)

| 種類 | メタ行の例 | 項目 |
|---|---|---|
| GitHub | 「GitHub · Python · ★ 1.2k · 更新 2023-11」 | 主要言語・スター数・最終更新年月 |
| YouTube | 「YouTube · 12:34 · 著者発表」 | 再生時間・属性 |
| スライド | 「iclr.cc · PDF · 24 枚」 | ドメイン・形式・枚数 |
| 解説記事 | 「zenn.dev · 解説記事 · 15 min」 | ドメイン・種別・読了目安 |
| Hugging Face | 「Hugging Face · Model · 12k downloads」 | repo種別・repo ID・downloads・likes・pipeline tag |
| プロジェクト | 「example.org · 公式プロジェクト」 | ドメイン・タイトル・OGP・検出元 |

**決定: 取得方法** — デザインには手段の記載がないため以下に確定する。

- GitHub: **GitHub REST API**(サーバーのアプリキー)。リポジトリ名・説明・主要言語・スター数・最終 push 日時を取得。
- YouTube: **oEmbed** でタイトル・サムネイル、**YouTube Data API v3** で再生時間。Data API が使えない場合は再生時間を省略して表示する(欠けても壊れない)。
- スライド(PDF): PDF をサーバーで取得しページ数をカウント(本体は保存しない)。タイトルは PDF メタデータ → なければ URL ファイル名。
- 解説記事: **OGP**(`og:title` / `og:image` / `og:site_name`)。読了目安は本文テキスト抽出から語数ベースで推定(日本語 600 字/分・英語 250 語/分を目安)。
- **取得失敗時**: URL のみでカードを作成し(タイトル= URL)、メタ行に取得不可の旨を控えめに表示。手動でタイトルを編集できる(§7)。取得失敗で追加自体を失敗させない(P3)。
- **決定**: メタは追加時に 1 回取得したスナップショットとし、自動更新はしない(スター数等の鮮度より API コストと安定表示を優先)。カードメニューから手動で再取得できる。
- YouTube の「著者発表」属性は自動判定しない(チャンネル名と著者名の照合は誤りやすい)。**決定**: v1 ではユーザーがメモまたはタイトルで補う運用とし、属性フィールドの自動付与は行わない。

## 4. ひとことメモ

- 各リソースに任意で 1 件の短文メモを添えられる。カード内に「💬」付きの淡色ボックスで表示する。
  - 例(GitHub): 「💬 train_reflow.py が §2.2 の手順に対応。データペアの再生成は generate_data.py」
  - 例(YouTube): 「💬 8:20〜 reflow の直感的な説明が分かりやすい」
- メモなしのカード(スライド/解説記事の例)はメモ欄を出さないコンパクト表示。
- **セクション参照チップ**: メモ本文に「§2.2」のような論文セクション参照チップを埋め込める。チップをクリックすると本文の該当セクションへジャンプする。アンカーの表現は共通アンカー仕様に従う([01-domain-model.md](01-domain-model.md)。ブロック単位ではなくセクション単位の参照で足りる)。
- **決定: 入力方法** — デザインに編集中状態は描かれていないため確定する。メモ欄(未入力ならカードメニューの「メモを追加」)クリックでインライン編集。編集中に `§` を入力するとセクション候補のサジェストを出し、選択でチップ化する。理由: 別ダイアログを開かず読書の流れを切らない(P6)。

## 5. 公式実装の自動検出

arXiv 論文の取り込み時に公式実装(GitHub リポジトリ)を自動検出し、リソースタブの最上部に **破線ボーダーの提案カード** で提示する。

- カード文言: 「**✦ 公式実装を検出しました** — arXiv ページのリンクから」+ 候補 URL(等幅フォント、例 `github.com/gnobitab/RectifiedFlow`)。
- アクション: 「**+ 追加**」(プライマリ)でリソースとして登録 / 「**無視**」で提案を却下しカードを消す。
- **勝手に追加しない**。破線=未確定の提案状態であり、確定はユーザーの 1 クリック(P6: 自動変更せず提案のみ)。
- 「+ 追加」で登録された GitHub リソースには緑系の「**公式実装**」バッジが付く。
- **決定: 提案カードはタブの件数バッジに数えない**。デザインで確定リソース 4 件+提案 1 件に対しバッジは「4」であることから確定。
- **決定: 「無視」は永続**(同一候補 URL を再提案しない)。却下した提案が再出現するのは P6 に反する。再取り込み時も無視リストを保持する。

**決定: 検出ロジック** — デザインの事実は「arXiv ページのリンクから」のみ。以下に確定する。

1. 取り込み時([02-ingest.md](02-ingest.md))に、(a) arXiv abs ページのリンク・コメント欄、(b) 構造化ドキュメントの `url` インライン(アブストラクト・脚注・第1セクションを優先)を走査し、`github.com/{owner}/{repo}` 形式の URL を抽出する。
2. 候補が複数ある場合は、アブストラクト/脚注での出現 → 出現回数の順で 1 件に絞る(提案は最大 1 件。並べて選ばせない。P2)。
3. 既に同一 URL のリソースが登録済み・無視済みの場合は提案しない。

### 5.1 Hugging Faceからの関連ソース収集(実装済み)

Hugging Face Paper PageはarXiv IDを入口に、関連するModel、Dataset、Space、GitHub実装、プロジェクトページを収集する(migration `0021_huggingface_resources`)。

取得にはHugging Face公開APIを使い、Paper PageのHTML構造には依存しない。

Paper APIの `githubRepo`、`projectPage`、`linkedModels`、`linkedDatasets`、`linkedSpaces` をResource候補へ変換する。

候補数はPaper Page 1件、GitHub 1件、project 1件、Model 5件、Dataset 3件、Space 3件を上限とする。

候補は `resource_links.status="suggested"` で永続化し、各候補を個別に採用または却下できるようにする。

採用前の候補はリソース件数へ数えず、却下後は同じURLを再提案しない。

Hugging Face URLを取り込み入口に使った場合は、URL内またはAPIレスポンスのarXiv IDを既存arXiv取り込みへ渡す。

取り込みに使ったHugging Face URL自体は確定Resourceとし、そこから見つけた関連リンクは候補とする。

### 5.2 GitHubコード対応解析(実装済み)

採用済みの公開GitHubリソースについて、論文の主張とコードのfile、symbol、行範囲の対応を生成できる(migration `0020_code_analysis`)。

設定 `code_analysis.mode` は `off`、`on_demand`、`automatic` の三値とし、既定を `on_demand` とする。

- `off`：新しい解析を実行しない。既存結果は閲覧できる。
- `on_demand`：GitHubカードの「コード対応を解析」から概算費用を確認して実行する。
- `automatic`：論文本文がreadyになり、高信頼の公式GitHub候補またはactive GitHubリソースが存在する時点で自動実行する。

automaticはHugging Face Paper APIの `githubRepo`、arXivメタデータから高信頼で検出した公式GitHub候補、採用済みまたは手動追加済みのactive GitHubリソースを対象とする。

公式根拠を持たないsuggestedリソースとdismissedリソースは解析しない。

公式候補を自動解析してもResource自体は自動採用せず、候補カードをsuggestedのまま保つ。

月額予算 `code_analysis.monthly_budget_usd` を持ち、既定を5.00 USDとする。

開始前にcommit SHA、対象ファイル数、token数、選択モデル、概算費用、予算残額を表示する。

予算を超える場合は外部APIを呼ばず、automatic jobは `waiting_budget` にする。

リポジトリは固定commitのarchiveとして取得するが、build、test、依存インストール、任意commandの実行は行わない。

解析結果は論文block anchorとGitHubの固定commit `#Lx-Ly` URLを結び、対応理由とconfidenceを表示する。

詳細設計は [Hugging Face関連ソース収集とGitHubコード対応解析](superpowers/specs/2026-07-17-huggingface-code-correspondence-design.md) を正とする。

## 6. リソースタブの UI(5a)

```
┌ サイドパネル(340px)────────────────┐
│ チャット | メモ | 注釈 6 | 図表 | リソース 4 | 情報 │ ← 排他タブ+件数
├──────────────────────────────────┤
│ [✦ 公式実装を検出しました …(破線)]  [+ 追加][無視] │
│ [GH] gnobitab/RectifiedFlow [公式実装]  開く ↗    │
│      GitHub · Python · ★ 1.2k · 更新 2023-11      │
│      💬 train_reflow.py が §2.2 の手順に対応 …    │
│ [▶] ICLR 2023 Oral — …             開く ↗        │
│      YouTube · 12:34 · 著者発表  [サムネ+12:34]   │
│ [PDF] 発表スライド(ICLR 2023)     開く ↗        │
│ [Z]  Rectified Flow を図で理解する  開く ↗        │
├──────────────────────────────────┤
│ [ URL を貼り付け — 種類を自動判定 ] [追加]         │
│ GitHub・YouTube・スライド・解説記事など。…        │
└──────────────────────────────────┘
```

- **カードの構成**(共通): 種類アイコン(26px 角)+ タイトル + メタ行 + 「開く ↗」。GitHub のタイトルは `owner/repo` を等幅フォントで表示。
- **種類別レイアウト**: GitHub / YouTube はひとことメモ欄付きの多段カード、YouTube はさらにサムネイル(高さ 96px、右下に再生時間バッジ)を持つ。スライド / 解説記事はメモがなければ 1 行構成のコンパクトカード(メモを付ければメモ欄が出る)。
- **「開く ↗」**: 外部リンクを新規タブで開く。リソースの閲覧は常に外部で行う(埋め込みプレーヤー等は持たない)。
- **フッター**: URL 入力欄+「追加」ボタン+ヘルプテキスト(§3 冒頭の逐語)。ここが唯一の手動追加経路。
- 一覧はタブ内で縦スクロール。提案カード(あれば)が常に最上部。

## 7. 編集・削除・並び順

デザインのフレーム内に編集・削除 UI は描かれていないため、以下を **決定** とする(操作は控えめに、カード単位のオーバーフローメニューに収める)。

- カードホバーで「⋯」メニューを表示: **メモを編集 / タイトルを編集 / 種類を変更 / メタを再取得 / 削除**。
- 削除は確認なしで実行し、直後に「元に戻す」トーストを数秒表示する(誤操作リカバリ)。
- 「公式実装」バッジは自動検出経由(§5)の登録にのみ付与し、手動での付け外しは v1 では提供しない(バッジの意味= システムが検出根拠を持つこと、を保つため)。
- **並び順は追加日時の昇順(新規が末尾)、提案カードのみ常に先頭**。理由: フッター入力で追加した直後に入力欄の直上へ現れ、視覚的連続性がある。手動並べ替え・種類別グルーピングは v1 では持たない(1 論文あたりのリソースは少数想定)。

## 8. データモデルとプライバシー

- エンティティ **ResourceLink**([01-domain-model.md](01-domain-model.md) 参照): `LibraryItem` 配下に置く。フィールド要点: `kind(github|youtube|slides|article|huggingface|project)`、`url`(正規化済み)、タイトル、ソースドメイン/サービス名、種類別メタ(JSON)、サムネイル参照、ひとことメモ(セクション参照チップ含む)、`official`(公式実装フラグ。自動検出提案の「+ 追加」でのみ true になる)、追加日時。無視した検出候補は無視リスト(候補 URL)として同じく LibraryItem 配下に持つ。
- **リソースは個人所有**。同じ論文を読む他ユーザーには見えず、コレクション共有ページ(4c)にも出さない(共有ページは書誌+要約+許可メモのみ)。アカウント削除時は LibraryItem ごと削除される。
- **決定**: 論文単位の Markdown エクスポート([06-library.md](06-library.md))にリソース一覧(種類・タイトル・URL・ひとことメモ)を含める。理由: ロックインしない(P5)。メモに埋めた §チップはテキスト表記(`§2.2`)に落とす。
- メタ取得はサーバーが行い、ユーザーのブラウザから対象サイトへ直接アクセスさせない。外部 API(GitHub / YouTube)のキーはサーバー管理で、レート制限時はメタなし登録に縮退する([09-nonfunctional.md](09-nonfunctional.md))。

## 9. 受け入れ基準

- [ ] フッターに GitHub / YouTube / PDF / 任意ブログの URL を貼ると、それぞれ `github` / `youtube` / `slides` / `article` として自動判定・登録される
- [ ] 判定不能な URL が「解説記事」として登録され、あとから種類を変更できる
- [ ] 登録後にタイトルと種類別メタ(言語・スター・更新 / 再生時間 / 枚数 / 読了目安)が自動表示され、取得失敗時も URL のみで登録が完了する
- [ ] YouTube カードにサムネイルと再生時間バッジが表示される
- [ ] ひとことメモに「§2.2」チップを埋め込み、クリックで本文の該当セクションへジャンプできる
- [ ] arXiv 取り込みで公式実装が検出されると破線の提案カードが最上部に出て、「+ 追加」で「公式実装」バッジ付きリソースになり、「無視」で二度と再提案されない
- [ ] リソースタブの件数バッジが確定リソース数と一致する(提案カードは数えない)
- [ ] 「開く ↗」で外部リンクが新規タブで開く
- [ ] 同一 URL の二重登録が防止される
- [ ] リソースが共有ページ(4c)に一切表示されない
- [ ] 論文単位 Markdown エクスポートにリソース一覧(URL・メモ含む)が含まれる
- [ ] Hugging Face Paper PageからGitHub、project、Model、Dataset、Space候補が上限内で提示され、個別に採用・却下できる
- [ ] Hugging Face URLからarXiv IDを解決し、既存arXiv取り込みへ接続できる
- [ ] コード対応解析を `off` / `on_demand` / `automatic` から選べる
- [ ] on-demand解析は概算費用の確認後にだけ始まり、automatic解析は高信頼の公式GitHub候補またはactive GitHubリソースで始まる
- [ ] 月額予算を超えた解析が外部APIを呼ばず `waiting_budget` になる
- [ ] 対応結果から論文blockと固定commitのGitHub行範囲を相互に開ける
- [ ] リポジトリのコードが実行されず、path traversal、symlink、秘密ファイル、容量超過が拒否される
